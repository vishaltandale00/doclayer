Meta-harnesses

A series — by Akhil R & Vishal Tandale

What is a harness?

A harness is the shell around a model — the prompts, tools, memory, and UI that decide what the model sees and what it can do. Most teams treat the harness as fixed scaffolding.

The interesting move is to author the harness itself: an LLM emits a typed UI manifest per phase, and the editor reshapes around the task at hand.

Planning looks like a research scratchpad. Drafting collapses to one editor. Review becomes a diff. Publishing produces a reader-harness that adapts to each reader's prior.

bound to: engineer · code background

Meta-harnesses

bound to: researcher · ML background

If the model's harness is the policy's environment, then learning a harness is meta-learning: the outer loop optimizes the shell that constrains the inner policy.

Reader-harness ≈ amortized inference: the published artifact carries a small lookup binding that re-renders the manifest for each reader prior.

Trace events power offline eval. Pinned panels become a per-author preference summary that biases future Architect output.

The whole lifecycle in one continuous reel, ~2 minutes end-to-end.

0–10s · trajectory proposal

The architect drafts a 4-article series. Atoms — title, contributors, audience, deploy target — snap into a manifest. Watch the harness-strip light up as scope locks.

10–28s · distribution into two work-windows

The trajectory fans out into per-author lanes. Vishal’s window and Akhil’s window appear side-by-side — each structurally different, authored from each author’s prior.

28–55s · parallel drafting

Both writers type live. Three AI agents (research, adversarial, visual) fire from the orchestration rail into each lane. Inline augments surface in prose — citation badges, claim flags, diagrams.

55–80s · cross-collab handoff

Vishal finishes a section. The architect packs it as a review packet and dispatches it across the gutter. Akhil’s harness reshapes on receive — outline collapses, review rail expands.

80–95s · single-article lifecycle

The reel zooms into one article shipping end-to-end: plan → draft → review → publish. The lifecycle ribbon advances chip-by-chip along the top.

95–110s · publish

One button. Build sequence streams: pack atoms → render manifest → emit reader-harness shell → deploy → warm CDN. Build target is doclayer.live.

110–120s · reader-harness goes live

Readers with different priors arrive. The same atoms re-bind into structurally different pages. The doc has become the reader’s harness.
