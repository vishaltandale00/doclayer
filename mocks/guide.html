doclayer · how the demo works

doclayer is a meta-harness for collaborative writing. Two writers and three AI agents share a series; the editor reshapes itself per phase, per author, and per task. When the article ships, the same document grammar becomes a reader-harness — the page re-binds to each reader's prior knowledge.

The mocks below are static HTML with CSS animations and auto-play on load. This guide tells you what to watch and why.

scenario notes

What to watch for, scene by scene.

Each scenario has a small ? guide button in the lower-left that opens a timed-beat narration synced to the on-page animation. The summaries below are static.

scenario 00

Lifecycle reel

The doc, born and shipped in one continuous reel.

  • 0–10s · The architect proposes a trajectory of 4 articles. Atoms — title, contributors, scope, deploy target — snap into a manifest.
  • 10–28s · Trajectory distributes into two work-windows. Vishal claims one lane, Akhil another. Each window is structurally different.
  • 28–55s · Parallel drafting. Three AI agents fire from the middle column: research, adversarial, visual. Each surfaces inline augments in both lanes.
  • 55–80s · Cross-collab handoff. Vishal completes a section; the architect packs it as a review packet; Akhil's harness reshapes around it.
  • 80–105s · Single-article lifecycle zooms in: plan → draft → review → publish.
  • 105–120s · Publish click. Build sequence runs. Reader-harness goes live; readers arrive with different priors; the page re-binds.
proves: the whole system in one breath. Lifecycle is a single continuous mechanism, not five disjoint apps. open scenario →
scenario 01

Bootstrap

Empty workspace → grammar atoms snap into a layout.

  • 0–3s · A blank form. Title, contributors, audience, deploy target. Four fields type themselves in.
  • 3–4s · The architect composes a manifest. Atoms are emitted as a typed graph (each field becomes a node).
  • 4–6s · Atoms snap into a layout. Outline, evidence panel, voice sampler appear in place — not because a template was picked, but because the manifest grammar bound them.
  • 6–8s · The harness emerges. Topbar, harness-strip, status bar light up. The next phase chip ("plan") is highlighted on the lifecycle ribbon.
proves: the editor isn't a template — it's authored from the manifest. open scenario →
scenario 02

Planning, two cursors

Two-up planning — divergent shells over a shared outline.

  • 0–5s · Two panes, same outline tree. Vishal's pane shows an evidence panel scraped from shared links; Akhil's pane shows a tension tracker.
  • 5–10s · Both cursors land in the outline. A section is reordered; both views update with a graph delta, not a bulk re-render.
  • 10–16s · Akhil disagrees on a sub-heading. The tension tracker opens a row. Resolution flows through chat in the tension card itself.
  • 16–22s · Tension resolves. The outline locks the section. Phase chip advances toward "draft".
proves: two harnesses can disagree on layout while sharing one document. open scenario →
scenario 03

Drafting

Typing live, agents fire inline, the harness cycles intent.

  • 0–6s · Live typing in the editor. The harness-strip pip is "composing".
  • 6–10s · Research and adversarial agents fire inline augments — citation badges, contradiction nudges — directly in the prose.
  • 10–16s · The cursor stalls. The harness shifts intent state: three revision variants surface in the right rail.
  • 16–20s · A variant is accepted; the paragraph swaps in place; the voice sampler updates Akhil's density meter.
  • 20–22s · Voice fidelity nudge: density drifted toward Vishal's voice. A tiny chip suggests tightening; one click accepts.
proves: the harness has intent state. Same text, different surface for write vs. revise. open scenario →
scenario 04

Review

Dismissals feed back into lesson memory.

  • 0–4s · AI suggestion chips bloom inside review blocks. Some are accepted; most are dismissed.
  • 4–8s · Dismissed chips animate as particles flying toward a lesson card. The particle physics is the model literally learning.
  • 8–12s · The lesson is composed ("user dislikes hedge-words in §3.x"). Author approves it; it commits to memory.
  • 12–16s · §3.2 auto-tightens: hedge-words vanish in-place because the new lesson re-ran the same suggester with a different prior.
  • 16–22s · A comment thread resolves. Phase chip advances to "publish-ready".
proves: the model learns the writer's taste during review, not after. open scenario →
scenario 05

Publish

Preview cycles priors. One button ships to doclayer.live.

  • 0–6s · Reader-harness preview cycles through three priors (novice / practitioner / researcher). Same atoms, three binds.
  • 6–10s · Preflight checks tick green: voice fidelity, citation coverage, lesson-memory commit, build-target reachable.
  • 10–12s · Publish click. Button pulses; the build sequence panel expands.
  • 12–18s · Build steps stream: pack atoms → render manifest → emit reader-harness shell → deploy → warm CDN.
  • 18–22s · Live state. Share URL appears; a copy-button flash hints the article is shipped.
proves: publish is a build, not an export. The output is a harness, not a static page. open scenario →
scenario 06

Reader harness

The doc becomes the reader's harness.

  • 0–5s · A single hero column with a prior switcher: novice / practitioner / researcher.
  • 5–12s · The reader changes prior. The article re-binds sequentially — paragraphs, citations, callouts re-render in place from the same atoms.
  • 12–18s · Money shot: the column expands into three parallel reader views side-by-side. Same atoms; three structurally different renders.
  • 18–22s · Comments thread shows three readers asking three differently-pitched questions about the same atom — proof the harness is per-reader, not per-page.
proves: the article itself is a harness. The reader's browser binds against the grammar. open scenario →
scenario 07

Multiplayer, two harnesses

In-the-moment view. 02–05 are slices of this.

  • 0–5s · Two shells side-by-side. Vishal's shell is research-heavy; Akhil's is voice/density-heavy. Same document, different harnesses.
  • 5–12s · Both authors type in different sections. Cursors don't collide; the harnesses don't share layout — they share grammar.
  • 12–18s · Sync gutter between the two shells routes a cross-collab event: a comment from Akhil shows up in Vishal's review rail as a review packet.
  • 18–22s · Density meter for §3.2 climbs as Akhil tightens. Both shells reflect the same chain state, expressed differently.
proves: "multiplayer" doesn't mean "shared view" — it means shared chain, separate harnesses. open scenario →
scenario 08

Workstream across lanes

Series spine, two writer lanes, three AI agents.

  • 0–4s · The series spine across the top — 4 articles, current article highlighted, others queued.
  • 4–10s · Two writer lanes drop down from the spine. Each lane is a per-author task queue with its own intent state.
  • 10–14s · Orchestration rail in the middle — the three AI agents (research, adversarial, visual). Pulses light up as agents fire.
  • 14–18s · A research agent surfaces a citation into Vishal's lane. An adversarial agent flags a claim in Akhil's lane. A visual agent renders a diagram inline.
  • 18–20s · Counter strip ticks: words, citations, lessons, sections approved. The shared spine state updates.
  • 20–22s · The lanes thesis: the harness arranges the work, not just the chrome. Lanes are first-class.
proves: doclayer is a workstream, not just an editor. Agents and writers share lanes. open scenario →
scenario 09

The review loop, end-to-end

One full protocol cycle, ~30 seconds, both sides visible.

  • 0–4s · Vishal types §3.2 live. The AI inline-augments two clauses as he writes.
  • 4–7s · Vishal pauses. The architect packs the section as a review packet — a "packet pip" flies across the gutter.
  • 7–10s · Akhil's harness gets a notification. The intent state shifts: "incoming packet → review-mode".
  • 10–17s · Akhil's harness reshapes: outline collapses, review rail expands, the packet is bound into the editor with full provenance.
  • 17–22s · Akhil composes comments. A second packet — comments + dispositions — gets dispatched back.
  • 22–26s · Vishal's harness receives the comments. Intent state shifts to "iterate-mode". His editor surfaces a diff rail.
  • 26–30s · Vishal applies revisions. The loop closes. Both harnesses return to their original intents.
proves: the protocol is symmetric. Either side's harness reshapes on receive, not on send. open scenario →

concepts

Glossary.

The vocabulary that turns up across the scenarios.

harness
The runtime around the model. UI shell, intent state, agent routing, panel layout — everything the model isn't, but everything that lets it be useful.
meta-harness
A controller that authors controllers. doclayer is one: it composes per-phase, per-author, per-task harnesses from a document grammar.
document grammar
The typed atoms the architect emits. Title, section, claim, citation, lesson, comment. Atoms bind layouts; layouts don't define atoms.
lane
A per-user task queue. Each writer has one; the series spine sits above them. Agents fire into lanes from a shared orchestration rail.
AI agents
Three: research (citation surfacing), adversarial (claim-checking, contradiction-flagging), visual (diagrams, callouts). Each augments writers inline.
cross-collab
Review and comment routing between authors. The architect packs a section as a packet; the receiving harness reshapes on receive.
reader-harness
What ships. The published article is a tiny harness that re-binds the grammar against each reader's prior. Same atoms, structurally different page.
lesson memory
Dismissed suggestions and accepted edits compose into lessons. Lessons rerun the suggester with a new prior. The model learns the writer's taste.
